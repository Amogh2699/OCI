Video 8: Introduction to Virtual Cloud Network (VCN)

A VCN is a private, software-defined network in an OCI region, providing a secure, isolated environment for your resources. It is regional, highly available, and scalable.

VCN Building Blocks:

  • VCN Address Space (CIDR): A private IPv4 address block (e.g., 10.0.0.0/16) that defines the network size.
  • Subnets: Subdivisions of the VCN's address range where resources are launched. Can be Public (internet-reachable) or Private (not internet-reachable).

Video 9: VCN Routing

Route Tables: A set of rules that tells your VCN where to send traffic destined for outside the VCN. A rule contains a Destination CIDR Block and a Route Target (gateway).

How Routing Works:

  • Implicit Local Routing: Traffic between subnets inside the same VCN is routed automatically. No rules needed.
  • Explicit Routing Logic: For traffic leaving the VCN, the route table is consulted. OCI uses the most specific rule (Longest Prefix Match).

Connecting VCNs (VCN Peering):

  • Local Peering (Same Region): Uses a Local Peering Gateway (LPG) on each VCN for a point-to-point link.
  • Remote Peering (Different Regions): Uses a Dynamic Routing Gateway (DRG) on each VCN.

Transit Routing with the DRG: The modern DRG can act as a centralized hub to connect many VCNs, simplifying architecture and avoiding a complex mesh of LPGs.

Video 10: VCN Security

Mechanism 1: Security Lists (SLs)

A virtual firewall at the subnet level. Rules apply to all instances in the subnet. Rules are stateful by default and use CIDR blocks for source/destination.

Mechanism 2: Network Security Groups (NSGs)

A more granular firewall applied to a group of VNICs (instances). The source/destination of a rule can be another NSG, allowing for application-centric security design.

Key Takeaway: For traffic to be allowed, it must be permitted by both the subnet's Security List AND any NSGs the instance is a part of.

Video 11: Load Balancer Service & Demo

A load balancer distributes traffic across multiple backend servers to provide High Availability (HA) and Scalability.

OCI Load Balancer Types

  • HTTP Load Balancer (Layer 7): Understands HTTP/S. Enables intelligent, content-based routing. Can be Flexible (custom bandwidth) or Dynamic (predefined shape).
  • Network Load Balancer (Layer 4): Understands TCP/UDP/ICMP. Offers extreme speed and ultra-low latency.

How to Choose: Use the HTTP LB for routing intelligence and advanced features. Use the Network LB for maximum performance and non-HTTP traffic.

Load Balancer - Demo Walkthrough

Scenario: A public load balancer distributing traffic to two web servers in a private subnet.

  1. Part 1: Network Foundation: Use the VCN Wizard to create a VCN with public and private subnets. Add an ingress rule to the Default Security List to allow HTTP traffic (port 80) from the internet (0.0.0.0/0).
  2. Part 2: Backend Servers: Create two compute instances (`web-server-1`, `web-server-2`) in the private subnet. Use a startup script to install Apache and create a unique `index.html` file on each.
  3. Part 3: Configure Load Balancer: Create a public Layer 7 Load Balancer. Place it in the public subnet. Configure the backend set with a Weighted Round Robin policy, add the two web servers, and set up an HTTP health check on port 80. Configure a listener for HTTP traffic on port 80.
  4. Part 4: Testing: Copy the load balancer's public IP. Pasting it into a browser and refreshing shows the content alternating between "My name is web server 1" and "My name is web server 2", confirming it works.
