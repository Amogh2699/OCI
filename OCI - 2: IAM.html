Video 3: OCI Identity & Access Management (IAM) - Introduction

1. The Two Pillars of IAM: Authentication & Authorization

Authentication (AuthN): Who are you? This is the process of verifying identity. It answers the question: "Is this person (or service) really who they claim to be?" Example: Providing a username and password to log in.

Authorization (AuthZ): What can you do? This is the process of granting permissions. It answers the question: "Now that we know who you are, what actions are you allowed to perform?" Example: A user is authorized to create new virtual machines but not to delete databases.

2. Core IAM Components

Identity Domain: A container for your users and groups. Think of it as the master directory for a specific user population and all its security settings.

Principals (The "Who"): An entity that is allowed to interact with OCI resources. Main principals are Users and Instance Principals.

Groups: A collection of users who all need the same type of access.

Compartments: Logical folders or containers to group related resources for organization and isolation.

3. The IAM Workflow
  1. Create an Identity Domain: Acts as the directory for your users.
  2. Create Users and Groups: Organize users into logical groups (e.g., NetworkAdmins, DatabaseAdmins).
  3. Place Resources in Compartments: Launch resources into specific, organized compartments.
  4. Write Policies: Connects everything by granting a group permissions on resources within a compartment.

Example Policy: `Allow group 'NetworkAdmins' to manage virtual-cloud-networks in compartment 'Production'`

4. Deep Dive: The Oracle Cloud ID (OCID)

Every resource in OCI gets a unique OCID, essential for CLI/SDK usage.

Syntax: `ocid1....`

  • ocid1: Version string.
  • resource-type: e.g., `instance`, `volume`, `tenancy`.
  • realm: A collection of regions (e.g., `oc1` for commercial).
  • region: Short code (e.g., `us-ashburn-1`). Global resources have no region here.
  • unique-id: The unique identifier string.

A compartment is a logical container used to group and isolate your cloud resources. Your tenancy comes with a top-level **Root Compartment**.

Key Characteristics & Rules
  • Single Home for Resources: A resource can only exist in one compartment at a time.
  • Cross-Compartment Interaction: Resources in different compartments can communicate with each other.
  • Resources are Movable: You can move most resources from one compartment to another.
  • Global Scope: Compartments are global and visible in every region of your tenancy.
  • Nested Hierarchy: Compartments can be nested up to six levels deep.

Video 5 & 6: IAM Authentication, Authorization & Demo

1. Authentication (AuthN): How Principals Prove Identity

A principal is an IAM entity allowed to act (Human Users, Resource Principals).

  • Username & Password: For interactive console sessions.
  • API Signing Keys: An RSA key pair for programmatic access via CLI/SDKs.
  • Authentication Tokens (Auth Tokens): A long-lived token string for third-party APIs.
2. Authorization (AuthZ): Defining Permissions with Policies

Policies are human-readable, deny by default, and attached to a tenancy or compartment. They always grant permissions to groups.

Syntax: `Allow group to in `

Verbs (Least to Most Privilege):

  • `inspect`: Read resource metadata.
  • `read`: `inspect` + read resource configuration.
  • `use`: `read` + work with existing resources (start/stop).
  • `manage`: `use` + full create, update, and delete permissions.
3. Demo: Authentication & Authorization in Action

Part 1: Authentication (AuthN) in Action

  1. Create User and Group: A new user `ociadmin` is created in a custom `Sandbox-Domain`. A new group `OCI-Admin-group` is created, and the user is added to it.
  2. User Activation: The user receives an activation email, clicks the link, and sets a new password.
  3. First Login: The user logs in with their tenancy, identity domain, username, and new password. This completes authentication.

Part 2: Authorization (AuthZ) in Action

  1. "Authenticated but Not Authorized": After logging in, the user tries to view resources and gets an authorization error. This shows OCI's "deny by default" model.
  2. Writing a Policy: An administrator uses the Policy Builder to grant the `OCI-Admin-group` permissions on the `sandbox` compartment. The generated policies are:
    • `Allow group Sandbox-Domain/OCI-Admin-group to manage buckets in compartment sandbox`
    • `Allow group Sandbox-Domain/OCI-Admin-group to manage objects in compartment sandbox`
  3. Testing Permissions: The `ociadmin` user refreshes. They still can't access Compute, but they can now successfully create a storage bucket inside the `sandbox` compartment. They are correctly denied from creating a bucket in the root compartment.

Video 7: OCI Best Practices - Tenancy Setup

The core principle is to delegate administrative responsibilities rather than using the default tenancy administrator for daily tasks.

The Three Foundational Best Practices

  1. Delegate Administration: Avoid using the root tenancy administrator account. Create dedicated administrative users and place them in an admin group (e.g., `OCI-Admins`).
  2. Use Dedicated Compartments: Isolate and organize your resources in a logical hierarchy. Do not place resources directly in the Root Compartment.
  3. Enforce Multi-Factor Authentication (MFA): Add an extra layer of security requiring something you know (password) and something you have (authenticator app).
